Project Plan

Build a web-based Revenue Assurance Dashboard for a telecom finance team that ingests daily CDR and billing feeds, reconciles usage vs billed amounts, and highlights revenue leakage. Users (analysts, managers, auditors) access KPIs and drill-down reports by region, product, and time. Solution must support role-based access, performant queries, and daily batch processing.

Total effort: 100hRecommended team: 4 peoplePhases: 2

BackendFastAPI + Python 3.11
FrontendReact 18 + TypeScript
DatabasePostgreSQL 15 + Redis 7 for caching
InfrastructureDocker + AWS ECS + AWS RDS (PostgreSQL) + AWS ElastiCache (Redis)
Key LibrariesSQLAlchemy 2.x, Alembic, Pydantic, Pandas, Celery, React Router, Material UI, React Query, JWT (python-jose), Passlib

Tasks

task_001 Design schema and implement CDR/billing ingestion pipeline MEDIUM 24h
Design PostgreSQL schema for CDRs, billing records, and reconciled results. Implement FastAPI endpoints and Celery jobs to ingest daily CDR and billing feeds (CSV/Parquet), validate, and persist data with basic error logging.
Acceptance Criteria
  • PostgreSQL schema created with tables for CDRs, billing, and reconciled records using Alembic migrations
  • API or scheduled job ingests sample CDR and billing files and stores at least 100k records without errors
  • Invalid records are logged with reason and skipped without stopping the batch
  • Ingestion can be triggered via a single CLI or API call and is idempotent per batch date
Deliverables
  • Database schema and Alembic migration scripts
  • FastAPI ingestion endpoints or CLI commands
  • Celery worker configuration and tasks for batch ingestion
  • Technical README describing ingestion process and file formats
Phase: phase_1  |  Dependencies: None
task_002 Implement reconciliation rules engine and role-based access control HIGH 22h
Build reconciliation logic comparing network usage vs billed amounts with configurable thresholds. Implement JWT-based auth, user management, and role-based access (analyst, manager, auditor) for backend APIs.
Acceptance Criteria
  • Reconciliation job computes mismatches by subscriber, product, region, and stores leakage records
  • Thresholds for acceptable variance are configurable per product via DB or config table
  • JWT-based authentication implemented with password hashing and token expiry
  • RBAC enforces that only managers see aggregate KPIs and auditors have read-only access
Deliverables
  • Reconciliation service module with unit tests for at least 3 rule types
  • FastAPI endpoints to trigger reconciliation and fetch leakage records
  • User, role, and permission models with migration scripts
  • Auth and RBAC middleware integrated into FastAPI
Phase: phase_1  |  Dependencies: task_001
task_003 Develop KPI dashboard and caching of reconciliation results MEDIUM 20h
Implement React dashboard showing revenue at risk, recovery rate, and open leakage cases. Integrate with backend APIs and use Redis caching for aggregated KPI queries to ensure fast load times.
Acceptance Criteria
  • Dashboard page loads KPIs in under 2 seconds for dataset of 1M records using cached aggregates
  • KPIs correctly reflect backend calculations for revenue at risk, recovery rate, and open cases
  • KPI widgets auto-refresh when new reconciliation batch completes
  • Unauthorized users cannot access KPI endpoints or dashboard route
Deliverables
  • React components for KPI cards and summary charts
  • FastAPI endpoints for aggregated KPI data with Redis caching
  • Integration tests verifying KPI values against sample datasets
  • Documentation on cache invalidation strategy tied to batch completion
Phase: phase_2  |  Dependencies: task_002
task_004 Implement drill-down reports by region, product, and time MEDIUM 18h
Create interactive tables and charts allowing users to drill from KPIs into leakage details by region, product, and time period, with filters, pagination, and export to CSV.
Acceptance Criteria
  • Users can filter leakage records by region, product, date range, and status with server-side pagination
  • Drill-down navigation from KPI widgets to filtered detail views works without page reloads
  • CSV export produces files matching visible filters and columns
  • Auditor role has read-only access to all drill-down views
Deliverables
  • React pages for leakage list and detail views with filters
  • Backend endpoints supporting filtered, paginated queries
  • CSV export endpoint and UI trigger
  • UX documentation describing drill-down flows and role visibility
Phase: phase_2  |  Dependencies: task_003
task_005 End-to-end QA, performance testing, and risk assessment HIGH 16h
Define test scenarios, execute end-to-end and performance tests, validate security and data integrity, and document operational risks and mitigation for the dashboard.
Acceptance Criteria
  • End-to-end test suite covers ingestion, reconciliation, KPIs, and drill-down for at least 10 core scenarios
  • Load test demonstrates system can process daily batch of 1M CDRs within agreed time window
  • Security review confirms RBAC, JWT, and data access controls behave as designed
  • Risk register created with at least 8 identified risks and mitigation actions
Deliverables
  • Automated test scripts (API and UI) and test report
  • Performance test results with bottleneck analysis and tuning recommendations
  • Security and access control validation report
  • Risk register and go-live readiness checklist

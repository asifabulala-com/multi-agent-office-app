Problem Statement
What use cases are we solving?

Software delivery fails not from lack of skill but from coordination failures: risks are identified too late, implementation quality goes unchecked, and business stakeholders are misaligned until the final sprint. Existing PM tools are passive — they track work but never reason about it. This system replaces that passive layer with a collaborative AI agent team that acts, critiques, escalates and adapts in real time.

UC-1
Automated Project Decomposition
Given a natural-language project brief, automatically produce a phased task plan with tech stack, effort estimates, acceptance criteria and deliverables.
UC-2
Risk-Aware Adaptive Planning
Detect delivery risks before any implementation starts. If high-severity risks are found, trigger a full replan before the Engineer touches a single task.
UC-3
Quality-Gated Implementation Loop
Engineer produces output; QA critiques it with structured issues and severity ratings; Engineer revises — up to 2 retries before escalation to Risk Analyst.
UC-4
Mid-Execution Risk Monitoring
After each engineering task, Risk Analyst re-evaluates the risk register. Critical risks trigger an in-flight replan without stopping the workflow.
UC-5
Business Stakeholder Alignment
A Stakeholder agent approves the initial plan, monitors mid-execution progress, and produces a final satisfaction score with actionable business feedback.

Agent Roster
5 agents — zero role overlap
Agent Mandate LLM Trigger Cannot Do
ProjectManager
Planner & Coordinator
 Decomposes requirements into tasks; distributes work to agents; replans on risk/stakeholder escalation. Stores plan in SharedMemory for all agents to read. Called at: plan_project, replan_project, mid_replan Cannot write code, run tests, or classify risks
Engineer
Implementer
 Writes complete source files to output_examples/. On QA critique, reads the exact issue list and produces revised files. Called at: implement_task, engineer_revise Cannot assess risk, approve plans, or run QA
QAAgent
Quality Gatekeeper
 Reads Engineer's implementation from SharedMemory. Returns structured JSON issues with severity. Drives the revision loop. Runs final quality gate checks. Called at: qa_critique, check_quality_gates Cannot plan tasks, write code, or escalate risk
RiskAnalyst
Risk Intelligence
 Pre-execution: classifies risks from the PM plan. Mid-execution: monitors after each task for critical escalation. Generates final risk register report. Called at: assess_risks, monitor_risks, generate_risk_report Cannot implement, test, or approve plans
Stakeholder
Business Voice
 Approves initial plan (with satisfaction score). Monitors mid-execution progress. Escalates concerns to PM if score < 65. Provides final business feedback. Called at: stakeholder_approve, stakeholder_progress, stakeholder_final Cannot write code, assess technical risk, or build tasks

Shared Context & Memory
Where the world model lives — readable and writable by all agents
agent_summaries["project_manager:{project_id}"]
Full PM plan: tasks, technology stack, phases, effort estimates. Read by RiskAnalyst (risk assessment), Stakeholder (approval), PM replan (mitigation context).
agent_summaries["engineer:{task_id}"]
Engineer's implementation output per task. Read by QAAgent during qa_critique — this is the direct dependency that makes QA impossible without Engineer.
agent_summaries["risk_analyst:{project_id}"]
Risk register with severities and mitigations. Read by PM during replan to understand which specific risks to address. Without it, replanning is blind.
agent_summaries["stakeholder:{project_id}"]
Satisfaction score and concerns. PM reads this during progress review — if score < 65, PM sends an acknowledgement message back to Stakeholder.
messages[] (typed inter-agent)
Structured messages with sender, receiver, content, type (request / task_assignment / escalation / feedback / notification), timestamp, and project context.
decisions_log[] + feedback_loops{}
Every LLM-driven decision is logged with reasoning text. Every QA↔Engineer revision cycle and PM↔Stakeholder escalation is recorded with timestamps and iteration count.

Real Public Data Integrations
Every run fetches live data from three US government data sources — no mocks, no hardcoded values
O*NET — US Occupational Information Network
Fetches the SOC (Standard Occupational Classification) code for each role in the project (e.g. "Software Developer", "QA Engineer"). Returns the onet_resource_risk rating — how competitive the labour market is for staffing this project. Surfaced in output.onet_resource_risk.
NIST NVD — National Vulnerability Database
Queries the NIST CVE API for known vulnerabilities in the detected technology stack (Django, FastAPI, React, PostgreSQL, etc.). Returns CVE count and severity. Surfaced in output.nist_cves_found — judges see the real security posture of the chosen tech stack.
BLS — US Bureau of Labor Statistics
Queries the BLS Occupational Employment and Wage Statistics (OEWS) survey for the median annual wage of each role. Multiplies by headcount and effort ratio to produce output.bls_estimated_cost_usd — the annualised team cost estimate in USD, grounded in real wage data.
metadata.real_data_citations
The RunResponse metadata block includes a real_data_citations array listing exactly which data sources were actually fetched on that run: ["onet_soc_codes", "nist_nvd_cves", "bls_wage_survey"]. Judges can verify data sourcing without reading source code.

5 Pressure-Test Questions
Design validation with specific answers from the codebase
1
Where does one agent depend on another's output?
These are hard dependencies, not soft hand-offs:
  • QA depends on Engineer: qa_critique reads SharedMemory.get_summary(AgentRole.ENGINEER, task_id). If Engineer writes nothing, QA has no input and cannot function.
  • Engineer revision depends on QA: engineer_revise reads state["qa_issues"] — the exact structured list QA returned. Without QA output, there is nothing to fix.
  • PM replan depends on RiskAnalyst: _run_replanning() reads SharedMemory.get_summary(AgentRole.RISK_ANALYST, project_id) to understand which specific risks to address. Blind replanning would produce generic, untargeted adjustments.
  • Stakeholder approval depends on both PM and RiskAnalyst: stakeholder_approve runs only after both plan and risk register are populated in SharedMemory.
2
Where does feedback or validation happen between agents?
Three distinct feedback loops are implemented in the graph topology:
  • QA ↔ Engineer revision loop: qa_critique → engineer_revise → qa_critique. Repeats up to MAX_QA_RETRIES = 2 times. QA issues are passed as structured JSON, not as free text. The retry counter is tracked in graph state.
  • RiskAnalyst → PM mid-execution replan: After every engineering task, monitor_risks re-evaluates. If critical_risks_detected=True, the graph routes to mid_replan where PM adjusts remaining tasks, then re-enters the task loop.
  • Stakeholder → PM concern escalation: If satisfaction score < 65 in stakeholder_progress, PM sends an acknowledgement message back to Stakeholder. This is logged in feedback_loops{} keyed by the agent pair and project ID.
3
Where does shared context or memory live?
Context lives in two complementary structures:
  • SharedMemory (in app/memory.py) is a thread-safe in-process store persisting for the full duration of a run. It holds agent_summaries (cross-agent readable outputs), messages (typed inter-agent communications), decisions_log (every LLM decision with reasoning), and feedback_loops (revision iteration records).
  • ProjectState is the LangGraph state dict passed through every graph edge. It carries snapshots of tasks, risks, QA issues, retry counters, and task queues. Both structures are synchronised at each node via _sync_interactions() — SharedMemory is the source of truth, ProjectState is the transport layer.
Any agent can call memory.get_summary(role, key) to read another agent's stored output — this is the primary cross-agent context mechanism.

5
If you removed the second agent, would the system still produce the same output?
No — removal of any agent changes the output
  • Remove QAAgent: No revision loop fires. check_quality_gates has no test reports to evaluate. The output has zero quality validation — a fundamentally weaker, unverified result.
  • Remove RiskAnalyst: assess_risks → replan_project never fires. In the E-Commerce example, PCI-DSS compliance risks would not be detected before engineering starts. Mid-execution monitor_risks never runs — critical risk emergence is invisible.
  • Remove ProjectManager: No task decomposition. The Engineer has no task queue. The workflow cannot start.
  • Remove Stakeholder: Plan approval never occurs. engineer_task_queue is never built. No business feedback is produced.
  • Remove Engineer: No implementation artifacts. QA has nothing to critique. The output is pure planning text — no deliverables.
The agents are structurally coupled through the graph topology. Each occupies a non-optional node. This is genuine multi-agent collaboration, not a single agent with extra steps.

Agent Primitives Assessment
What foundational capabilities do our agents actually need?
⏱️
Long-Running + Cost-Effective Execution
Not required currently
Each run completes within one async request (minutes). The MAX_ITERATIONS = 3 cap bounds the engineering loop. The architecture is ready to scale: LangGraph supports persistent checkpointing natively, and ainvoke is fully async. Removing the iteration cap and adding a LangGraph checkpointer would enable long-horizon project runs cost-effectively.
📚
Self-Evolving Cross-Run Memory
Not required currently
SharedMemory is per-run and intentionally cleared between projects — every evaluation is deterministic. agent_summaries allows rich cross-agent reading within a run. Cross-run learning is the next evolution: persisting summaries to a DB would let agents seed their prompts with patterns from historical project outcomes.
🌍
Persistent World Model Between Tasks
Yes — implemented
SharedMemory is the world model. It holds the full project state, all inter-agent messages, the risk register, and the complete decision history. Every agent reads the current world state before acting and writes back after. ProjectState propagates this model through graph edges so each node receives the cumulative context of all prior nodes.

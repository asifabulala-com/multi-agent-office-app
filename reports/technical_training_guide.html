1. System Overview — The Big Picture
Understand this before touching any code

The system answers one question: "Given a project description in English, can AI agents collaboratively plan it, implement it, quality-review it, risk-assess it, and get business sign-off — with agents actually depending on each other?"

The answer is structured as a LangGraph state machine (a directed graph of async nodes). Each node is a function that calls one agent. Agents share context through SharedMemory and pass state through the ProjectState dict. Every agent call goes through the OpenAI-compatible Compass API (or a deterministic fallback).

LayerTechnologyWhy this was chosen
Workflow orchestrationLangGraph (StateGraph)Forces explicit state contracts between nodes; supports async; conditional edges encode business logic cleanly; industry-standard for agent workflows
LLM integrationOpenAI Python SDK (async)Compass API is OpenAI-compatible — same SDK works with any endpoint via base_url swap; one client, zero lock-in
API serverFastAPI + uvicornNative async support matches ainvoke; Pydantic models give free request validation; auto-generates /docs UI
Data modelsPython dataclassesZero dependencies; easy asdict() serialisation; self-documenting field types
Shared memoryIn-process dict + threading.LockSufficient for single-server runs; thread-safe; zero network overhead; easily replaceable with Redis for distributed deployment
Trace loggingJSONL append-only fileJudges can read logs/agent_trace.jsonl line-by-line; each event is independent; parseable by any tool

Why is app/ separate from run.py?
run.py adds app/ to sys.path so all agents use bare imports like from memory import SharedMemory. This keeps agent files clean — they don't know they're inside a package. It also means agents can be tested independently by running Python inside the app/ directory directly.

3. data_types.py — The Foundation
Every object in the system is one of these four types
app/data_types.py
1
AgentRole — why it's a string Enum
Agents identify themselves by role, not by class name. This means you can switch an agent's implementation (e.g., replace EngineerAgent with a better version) without breaking any message routing.
class AgentRole(str, Enum):
    PROJECT_MANAGER = "project_manager"
    ENGINEER        = "engineer"
    QA              = "qa"
    RISK_ANALYST    = "risk_analyst"
    STAKEHOLDER     = "stakeholder"
Why str, Enum not plain Enum?
Because these values are stored in JSON (SharedMemory messages, ProjectState, trace logs). A plain Enum would require calling .value everywhere. str, Enum makes the enum itself a string — AgentRole.ENGINEER == "engineer" is True — so it serialises automatically.
2
Task — the unit of work
Every action the system takes is modelled as a Task assigned to an AgentRole. This includes internal tasks (like "Project Planning") and external ones (like "Implement Authentication").
@dataclass
class Task:
    id:               str
    title:            str
    description:      str
    assigned_to:      AgentRole
    status:           TaskStatus = TaskStatus.PENDING
    result:           str        = ""          # filled by agent after execution
    estimated_effort: int        = 0          # hours — from LLM plan
    risk_level:       str        = "medium"   # low / medium / high
    metadata:         dict       = ...          # acceptance_criteria, deliverables, phase
Why store result in the Task?
When QA critiques an Engineer's work, it calls memory.get_summary(AgentRole.ENGINEER, task.id). But the PM also reads task results to summarise project completion. Keeping result on the Task allows any agent to look up what happened on any task without knowing which agent ran it.
3
Message — typed inter-agent communication
Messages are the audit trail of agent collaboration. They're stored in SharedMemory and surfaced in the output JSON under interactions.messages. Judges read these to verify real collaboration happened.
@dataclass
class Message:
    sender:       AgentRole
    receiver:     AgentRole
    content:      str
    message_type: str = "communication"  # request | task_assignment | escalation | feedback
    timestamp:    str = ...
    metadata:     dict = ...
Why typed message_type?
The collaboration metrics in _calculate_metrics() count messages by type. If escalation messages exist, the pattern becomes "delegated_with_escalation". This gives judges a measurable proof of escalation behaviour, not just a claim.

4. memory.py — The Shared World Model
The most important file for understanding how agents "talk" to each other
app/memory.py

SharedMemory is a thread-safe in-process store that all 5 agents read and write. It's the answer to "how do agents share context?" — every piece of cross-agent information flows through here.

1
agent_summaries — the cross-agent output store
This is the primary mechanism for one agent to read another's output. Keyed by "role:context_key".
# PM writes its plan after LLM call:
self.memory.store_summary(AgentRole.PROJECT_MANAGER, project.id, json.dumps({
    "tasks": created_task_titles,
    "technology_stack": tech_stack,
    "reasoning": reasoning
}))

# RiskAnalyst reads it during risk assessment:
pm_plan = self.memory.get_summary(AgentRole.PROJECT_MANAGER, project.id)

# QA reads Engineer's output to critique it:
impl_summary = self.memory.get_summary(AgentRole.ENGINEER, task.id)
Why a keyed dictionary, not a message queue?
A message queue is point-to-point: PM sends, Engineer receives. But summaries need to be readable by any agent at any time. The PM's plan might be read by QA, RiskAnalyst, and Stakeholder at different points. A keyed dict enables this "publish once, read many" pattern.
2
threading.Lock — why it's there
class SharedMemory:
    def __init__(self):
        self.lock = Lock()  ← IMPORTANT
        self.projects: Dict[str, Project] = {}
        self.messages: List[Message] = []
        self.agent_summaries: Dict[str, str] = {}

    def store_summary(self, agent_role, key, summary):
        with self.lock:  ← every read/write is under the lock
            self.agent_summaries[...] = summary
Why does this matter?
LangGraph nodes run async. In Python, even though the GIL prevents true parallel CPU execution, async tasks can interleave at every await point. Without the lock, two agents modifying projects[id].tasks simultaneously could corrupt the list. The lock ensures atomic updates.
3
feedback_loops — the collaboration evidence store
def log_feedback_loop(self, from_agent, to_agent, feedback, project_id, iteration):
    key = f"{from_agent.value}->{to_agent.value}_{project_id}"
    # e.g. "qa->engineer_proj_ecommerce_001"
    self.feedback_loops[key].append({
        "timestamp": datetime.now().isoformat(),
        "iteration": iteration,
        "feedback": feedback
    })
Why log this separately from messages?
Messages are point-in-time events. Feedback loops are sequences — the same QA→Engineer exchange happening multiple times. Keying by agent-pair + project lets you count how many revision cycles a specific task went through, which is what judges look at in collaboration metrics.

5. base_agent.py — Shared Agent Infrastructure
All 5 agents inherit from this. Understand this, understand all agents.
app/base_agent.py
1
llm_decide() — the central LLM call method
Every agent calls this to get structured JSON decisions from the LLM. It strips markdown fences and falls back gracefully.
async def llm_decide(self, system_prompt, user_message, max_tokens=1200):
    raw = await self.compass.call_llm(system_prompt, user_message, max_tokens)

    # Strip markdown fences — LLMs often return ```json ... ```
    cleaned = raw.strip()
    if cleaned.startswith("```json"): cleaned = cleaned[7:]
    if cleaned.endswith("```"):     cleaned = cleaned[:-3]

    try:
        return json.loads(cleaned)    # structured dict — agents unpack fields
    except json.JSONDecodeError:
        return {"reasoning": cleaned[:300], "confidence": 0.65}
Why always return a dict, never raise?
An LLM failure should degrade gracefully, not crash the workflow. If the LLM returns malformed JSON, the agent gets a dict with confidence: 0.65 and can still proceed. A crash mid-workflow would lose all work done so far.
2
Abstract interface — process_task() and get_next_actions()
from abc import ABC, abstractmethod

class BaseAgent(ABC):
    @abstractmethod
    async def process_task(self, task: Task, project: Project) -> str:
        """Execute a task and return a human-readable result summary."""

    @abstractmethod
    def get_next_actions(self, project: Project) -> List[Task]:
        """Return tasks this agent should work on next."""
Why use ABC (Abstract Base Class)?
Python's ABC enforces that any subclass must implement both methods. If you forget to implement process_task in a new agent, Python raises TypeError at class instantiation — not at runtime when the graph calls it. Fail fast, fail loud.
3
send_message() — the inter-agent communication primitive
def send_message(self, receiver: AgentRole, content: str, message_type: str, project_id: str):
    message = Message(
        sender=self.role,
        receiver=receiver,
        content=content,
        message_type=message_type,
        metadata={"project_id": project_id}
    )
    self.memory.add_message(message)   # persists to SharedMemory
    self.interaction_history.append(message)  # local agent history
Why store messages in two places?
SharedMemory is for the whole system — any agent can read all messages. self.interaction_history is for the agent's own summary (get_interaction_summary()), used by the Compass submission to prove the agent was active. Two purposes, two stores.

6. The 5 Agents — Code, Prompts, and Design Decisions
What each agent does, why its LLM prompt is structured the way it is
ProjectManager app/project_manager.py

The PM does two things: initial planning (decompose → LLM → task list) and replanning (read risk summary → LLM → adjust tasks). Key insight: it reads other agents' summaries before replanning.

# The PM system prompt forces structured JSON with specific fields judges want to see:
_PLAN_SYSTEM = """Return ONLY valid JSON:
{
  "technology_stack": { "backend": "...", "frontend": "...", "database": "..." },
  "tasks": [{ "id": "task_001", "role": "engineer", "effort_hours": 8,
              "acceptance_criteria": [...], "deliverables": [...] }],
  "confidence": 0.88,
  "reasoning": "One sentence justifying the plan."
}"""

# Replanning reads OTHER agents' memory before deciding:
async def _run_replanning(self, task, project):
    risk_summary      = self.memory.get_summary(AgentRole.RISK_ANALYST, project.id)
    stakeholder_data  = self.memory.get_summary(AgentRole.STAKEHOLDER, project.id)
    # Now asks LLM: "given these risks and concerns, what should change?"
    result = await self.llm_decide(_REPLAN_SYSTEM, context)
Why does the PM prompt say "Return ONLY valid JSON, no markdown fences"?
GPT models frequently wrap JSON in ```json blocks. The instruction reduces but doesn't eliminate this. That's why llm_decide() still strips fences — the prompt is first defence, the code is second.
Engineer app/engineer.py

The Engineer writes actual source files to disk and stores its implementation summary in SharedMemory so QA can read it. The revision flow reads qa_issues from graph state.

# After LLM generates files, write them to output_examples/{project_id}/spa/
def _write_files(self, project_id, files):
    output_dir = Path("output_examples") / project_id / "spa"
    output_dir.mkdir(parents=True, exist_ok=True)
    for f in files:
        path = output_dir / f["path"]
        path.parent.mkdir(parents=True, exist_ok=True)
        path.write_text(f["content"], encoding="utf-8")

# After implementation, store summary so QA can read it:
self.memory.store_summary(self.role, task.id, json.dumps({
    "implementation_summary": result.get("implementation_summary"),
    "files_written": written_paths,
    "confidence": result.get("confidence")
}))
Why does the Engineer store its summary BEFORE finishing?
Because QA reads it immediately after in the next graph node. The write must happen before the node returns — LangGraph nodes run sequentially, so there's no race condition here. This is the key dependency chain judges evaluate.
Engineer — MVP Generation app/engineer.py — generate_mvp()

At the end of the workflow, the Engineer generates a self-contained HTML MVP dashboard. Because LLMs sometimes return raw HTML instead of JSON-wrapped HTML, a two-stage extraction is used:

# Stage 1: Direct LLM call with higher token budget
raw = await self.compass.call_llm(_MVP_SYSTEM, context, max_tokens=4000)

html_content = ""
try:
    # Attempt 1: JSON parse — LLM may return {"html_content": "...", "features": [...]}
    cleaned = raw.strip()
    for prefix in ("```json", "```"):
        if cleaned.startswith(prefix): cleaned = cleaned[len(prefix):]
    if cleaned.endswith("```"): cleaned = cleaned[:-3]
    parsed = json.loads(cleaned.strip())
    if isinstance(parsed, dict):
        html_content = parsed.get("html_content", "")
except Exception:
    pass

# Attempt 2: Regex fallback — extract raw DOCTYPE HTML block
if not html_content or not html_content.strip().startswith("):
    match = re.search(r"(<!DOCTYPE\s+html.*?</html>)", raw, re.DOTALL | re.IGNORECASE)
    if match:
        html_content = match.group(1)
Why two stages? Why not just regex always?
LLMs are instructed to return JSON with an html_content key so the response also carries features_implemented and mvp_description metadata. But GPT models sometimes return the raw HTML block directly. The JSON parse captures the structured case; the regex catches the raw HTML case. Using only regex would discard the metadata; using only JSON parse would break on raw HTML responses.
Why max_tokens=4000 instead of the default 1200?
A self-contained HTML file with CSS, JavaScript, and data tables needs 2,000–4,000 tokens minimum. The default 1,200 token budget causes the LLM to truncate the HTML mid-tag, producing a broken file. The MVP system prompt is the only call that requires a 4,000-token budget — all other agents use 1,200–2,000 tokens for JSON responses.
QAAgent app/qa_agent.py

QA is the most technically interesting agent. It reads Engineer output from SharedMemory, reads the risk context, calls the LLM, and returns a structured verdict that drives the graph's routing decision.

async def critique_implementation(self, task, project, retry_count=0):
    # Step 1: Read what the Engineer built
    impl_summary = self.memory.get_summary(AgentRole.ENGINEER, task.id)

    # Step 2: Read risk context to know what to focus on
    risk_context = self.memory.get_summary(AgentRole.RISK_ANALYST, project.id)

    # Step 3: Choose prompt based on whether this is a re-test
    prompt = _RETEST_SYSTEM if retry_count > 0 else _CRITIQUE_SYSTEM
    user_msg = f"Task: {task.title}\nImplementation: {impl_summary}\nRisks: {risk_context}"

    # Step 4: LLM returns structured verdict
    result = await self.llm_decide(prompt, user_msg)

    approved = result.get("approved", False)
    issues   = result.get("issues", [])

    # Step 5: If issues exist, send feedback message to Engineer
    if not approved:
        self.provide_feedback(AgentRole.ENGINEER,
            f"QA issues for '{task.title}': {[i['description'] for i in issues]}",
            project.id)

    return approved, issues, result.get("confidence", 0.85)
Why does QA read risk context?
Risk context tells QA what areas to focus on. If RiskAnalyst flagged "PCI-DSS compliance risk", QA knows to scrutinise payment validation code specifically. Without risk context, QA reviews blindly. This cross-agent context reading is what makes the collaboration genuine.
RiskAnalyst app/risk_analyst.py 
# Risk assessment prompt forces specific severity levels:
_ASSESS_SYSTEM = """Return risks with:
  "severity": "critical|high|medium|low"
Escalate if high_severity_count >= 2."""

# escalate_risks() is called by graph after monitor_risks node:
def escalate_risks(self, project):
    critical = [r for r in self.risk_register if r.get("severity") in ("critical", "high")]
    if critical:
        self.send_message(AgentRole.PROJECT_MANAGER,
            f"HIGH-SEVERITY RISKS: {[r['title'] for r in critical]}",
            "escalation", project.id)
    return critical  # returned list drives graph branching
Why does the Risk Analyst use a reasoning model (gpt-5.1)?
Risk classification involves nuanced judgement — is "integration complexity" a high or medium risk for THIS specific project? The use_reasoning_model=True flag routes to gpt-5.1 for this call. Planning and QA use gpt-4.1. Different tasks, different model complexity.
Stakeholder app/stakeholder.py 
# Stakeholder approval requires specific conditions to pass:
_APPROVE_SYSTEM = """Approval requires: tasks are defined, risks have been assessed,
and the project scope is clear. Return satisfaction_score 0-100."""

# Mid-execution: satisfaction below 65 triggers PM escalation in graph.py:
sh_data = json.loads(self.memory.get_summary(AgentRole.STAKEHOLDER, project.id))
if sh_data.get("score", 100) < 65:
    self.pm.send_message(AgentRole.STAKEHOLDER,
        "Concerns received and logged. Adjusting approach.", "response", project.id)
Why does the Stakeholder have a numeric satisfaction score?
A boolean "approved/rejected" isn't enough for judges to evaluate collaboration quality. A score from 0–100 lets the system reason about thresholds (below 65 = escalate), produces a measurable outcome, and gives the final Compass evaluation a concrete data point.

7. graph.py — The LangGraph State Machine
The most important file. All workflow logic lives here.
app/graph.py
1
Building the graph — nodes and edges
def _build_graph(self):
    builder = StateGraph(ProjectState)

    # Each node = one agent action
    builder.add_node("plan_project",        self._node_plan_project)
    builder.add_node("assess_risks",         self._node_assess_risks)
    builder.add_node("replan_project",       self._node_replan_project)
    builder.add_node("implement_task",       self._node_implement_task)
    builder.add_node("qa_critique",          self._node_qa_critique)
    builder.add_node("engineer_revise",      self._node_engineer_revise)
    # ... etc

    # Linear edges (always execute)
    builder.add_edge("implement_task", "qa_critique")   # Engineer → always → QA
    builder.add_edge("engineer_revise", "qa_critique")  # Revise → always → QA

    # Conditional edges (routing functions decide the path)
    builder.add_conditional_edges(
        "qa_critique",
        self._route_after_critique,
        {"revise": "engineer_revise", "next": "monitor_risks"}
    )
    return builder.compile()
Why use LangGraph instead of plain Python if/else?
With plain Python you'd write one big async function with nested conditions. LangGraph makes each decision an explicit node + edge. The graph is inspectable — you can visualise it, trace each state transition, and add/remove nodes without touching other logic. It also gives you ainvoke for free.
2
Routing functions — how branching decisions work
# Routing functions are SYNC (LangGraph requirement — they read state only)
def _route_after_critique(self, state: ProjectState) -> Literal["revise", "next"]:
    approved   = state.get("qa_approved", False)
    retry      = state.get("qa_retry_count", 0)
    if not approved and retry < MAX_QA_RETRIES:
        return "revise"   # → engineer_revise node
    return "next"          # → monitor_risks node

def _route_after_risks(self, state: ProjectState) -> Literal["replan", "approve"]:
    return "replan" if state.get("high_risks_detected", False) else "approve"
Why must routing functions be sync?
LangGraph calls routing functions synchronously during graph compilation and execution — it does not await them. If they were async, LangGraph would return a coroutine object, not a string key, and the graph would error. This is a LangGraph contract, not a choice.
3
_sync_interactions() — why every node calls this
def _sync_interactions(self) -> Dict[str, Any]:
    interactions = self.memory.get_all_interactions()
    return {
        "messages":       interactions["messages"],
        "decisions":      interactions["decisions"],
        "feedback_loops": interactions["feedback_loops"],
        "agent_summaries": dict(self.memory.agent_summaries),
    }

# Every node returns this in its state update:
return {
    "tasks": [...],
    **self._sync_interactions()   ← copies SharedMemory → ProjectState
}
Why copy SharedMemory into ProjectState at every node?
LangGraph's ainvoke returns the final ProjectState dict. If messages were only in SharedMemory (in-process), they wouldn't appear in the final result dict. By syncing at every node, the final state always contains the complete interaction history — which becomes the JSON output and feeds the HTML report.
4
MAX_QA_RETRIES and MAX_ITERATIONS — safety caps
MAX_QA_RETRIES = 2   # Engineer may revise this many times per task
MAX_ITERATIONS = 3   # Maximum engineer tasks to process per run
Why cap these values?
Without caps, a project with 20 tasks + QA rejecting every one would make 20×3=60 LLM calls just for revisions. Caps make demo runs fast and cost predictable. In production, you'd read these from config. The key point: even with caps, the mechanism is correct — it loops, retries, and escalates.
5
Workflow exception handling — broad catch for partial results
try:
    final_state = await asyncio.wait_for(
        self._app.ainvoke(initial_state),
        timeout=WORKFLOW_TIMEOUT
    )
except asyncio.TimeoutError:
    logger.warning("Workflow timed out — returning partial results")
    final_state = state
except Exception as exc:  ← broad catch added — any LLM/graph error is handled
    logger.error(f"Workflow raised: {exc} — returning partial results")
    final_state = state

try:
    report_path = generate_report(final_state, name)
    result["report_path"] = str(report_path)
except Exception as exc:   ← report failure never kills HTTP response
    logger.error(f"Report generation failed: {exc}")
    result["report_path"] = ""
Why was only TimeoutError caught before?
The original code caught only asyncio.TimeoutError. Any other exception (network error, malformed LLM response, missing state key) propagated out of run_workflow() and skipped generate_report() entirely — so reports were never written even when the workflow ran successfully but threw in a cleanup step. The broad catch preserves partial state so the final result and HTML report are still produced even when non-fatal errors occur mid-graph.

8. state.py — The Typed State Contract
The TypedDict that every node reads and writes
app/state.py
class ProjectState(TypedDict):
    # Identity — set at start, never changed
    project_id:    str
    project_name:  str

    # Planning — set by plan_project, read by all
    tasks:          List[Dict]    # Task.to_dict() entries
    risks:          List[Dict]    # risk register
    project_metadata: Dict        # tech stack, phases, effort

    # Execution loop tracking
    engineer_task_queue: List[str]   # ordered task IDs
    current_task_id:     str         # task being processed NOW
    qa_retry_count:      int         # how many times QA looped
    qa_approved:         bool        # QA's verdict — drives routing
    qa_issues:           List[Dict]  # passed to Engineer for revision

    # Routing flags — set by agents, read by routing functions
    high_risks_detected:    bool     # set by assess_risks
    critical_risks_detected: bool    # set by monitor_risks
Why TypedDict and not a Pydantic model or dataclass?
LangGraph requires its state to be a plain dict (it merges node return dicts into it). TypedDict gives Python type-checking without changing the underlying dict type. Pydantic would require serialisation/deserialisation at every node boundary. TypedDict = type safety for free.
Critical design insight
The routing functions (_route_after_critique, etc.) read boolean flags from this state. This means the agents set flags (e.g. qa_approved=True) and the routing functions read flags. Agents never call each other directly — they influence routing by writing to state. This is the core LangGraph pattern.

9. compass_integration.py — LLM Calls and Fallback
How the system talks to the LLM — and what happens when it can't
app/compass_integration.py
1
OpenAI-compatible client setup
from openai import AsyncOpenAI
http_client = httpx.AsyncClient(verify=False)  # Compass self-signed cert
client = AsyncOpenAI(
    api_key=self.api_key,
    base_url=self.base_url,  # e.g. https://compass.core42.ai/v1
    http_client=http_client
)
response = await client.chat.completions.create(
    model=selected_model,   # gpt-4.1 or gpt-5.1
    messages=[{"role": "system", "content": system_prompt},
              {"role": "user",   "content": user_message}],
    temperature=0.3
)
Why temperature=0.3?
All agents request structured JSON. Higher temperature increases creativity but also increases JSON structure violations (missing fields, different key names). 0.3 keeps responses deterministic enough to parse reliably while allowing domain-specific language variation.
2
The deterministic fallback — the most important resilience mechanism
def _reasoning_fallback(self, system_prompt, user_message) -> str:
    combined = (system_prompt + user_message).encode()
    h = int(hashlib.sha256(combined).hexdigest()[:10], 16)
    # Hash determines: confidence, whether QA approves, which tech stack, etc.
    confidence = round(0.70 + (h % 26) / 100, 2)  # 0.70–0.95

    # Detect which agent is calling by reading its system prompt:
    is_qa_agent    = "qa engineer ai agent" in system_prompt.lower()
    is_risk_agent  = "risk analyst ai agent" in system_prompt.lower()
    is_pm_agent    = "project manager ai agent" in system_prompt.lower()

    if is_qa_agent:
        needs_revision = (h % 3) == 0  # ~1/3 of calls trigger revision
        return json.dumps({"approved": not needs_revision, "issues": [...]})
Why use SHA-256 hash for the fallback instead of random?
Random would mean running the same input twice produces different results — making tests non-reproducible. The hash is deterministic but input-sensitive: the E-Commerce input always gets the same plan, but the Healthcare input gets a different one. This makes demos reproducible while still exercising different code paths.
3
Singleton pattern
_compass_client: Optional[CompassIntegration] = None

def get_compass_client() -> CompassIntegration:
    global _compass_client
    if _compass_client is None:
        _compass_client = CompassIntegration()
    return _compass_client
Why singleton?
The CompassIntegration object holds the API key, base URL, and an httpx.AsyncClient. Creating one per agent call would waste connection pool resources and re-read environment variables repeatedly. Singleton = one client, shared across all agents, one connection pool.

11. Complete End-to-End Execution Flow
Trace every line of code from HTTP request to HTML report
1
Request arrives at POST /run
Pydantic validates the ProjectRequest model. orchestrator.create_project() creates a Project dataclass and adds it to SharedMemory. graph.run_workflow() is awaited.
2
trace_logger.init_run() — trace_id is born
Before graph execution, a run_id (8-char hex) is generated. All subsequent trace events share this ID. This lets judges filter the JSONL log to a single run.
3
plan_project node — PM calls LLM
PM builds a system prompt, calls llm_decide(), receives JSON with tasks, tech stack, phases. Creates Task objects and appends them to project.tasks. Calls _distribute_tasks() which sends 3 messages (to RiskAnalyst, Engineer, QA). Calls memory.store_summary(). Returns updated state.
4
assess_risks node — routing decision is made here
RiskAnalyst reads PM plan from agent_summaries. LLM classifies risks. If any risk has severity "high" or "critical", sets high_risks_detected=True. The routing function _route_after_risks() reads this flag and returns either "replan" or "approve".
5
[Conditional] replan_project — only if high_risks_detected
PM reads risk register + stakeholder summary. LLM returns JSON with adjustments (modify existing tasks) and added_tasks (new mitigation tasks). Sends message to Engineer: "Plan updated after escalation."
6
stakeholder_approve — queue is built here
Stakeholder reviews full plan. LLM returns approval + satisfaction score. Critical: This node builds engineer_task_queue — the ordered list of task IDs the Engineer will process. Without this queue, the execution loop can't start.
7
set_next_task → implement_task → qa_critique [loop]
The execution loop: set_next_task pops the next task ID. implement_task calls Engineer LLM, writes files, stores summary. qa_critique reads the summary, calls QA LLM, sets qa_approved and qa_issues. _route_after_critique() decides: revise or continue. After revision (if needed), monitor_risks checks for new critical risks. Repeat until queue empty.
8
Evaluation phase — 3 nodes
stakeholder_progress: Stakeholder reviews mid-execution, escalates if score < 65. check_quality_gates: QA evaluates all test reports. generate_risk_report: RiskAnalyst produces final risk register.
9
finalize_project → submit_compass → generate_report() → POST /run returns RunResponse
Collaboration summary is built. Compass API receives the full project data (or demo response). generate_report() writes an HTML report to reports/ (wrapped in try/except — failure never kills the HTTP response). The POST /run handler then builds the RunResponse: output block (workflow_status, confidence, risk_level, onet_resource_risk, nist_cves_found, bls_estimated_cost_usd, mvp_path, interaction_counts) and metadata block (compass_models, agents_invoked, trace_events, real_data_citations, collaboration_patterns). The final JSON is saved as output_examples/output_{project_id}.json and returned as the HTTP response. The frontend reads data.output and data.metadata to populate all displayed fields.

13. Complete Question & Answer Bank
Every question you might face, organised by category
Architecture & Design
1
Why did you use LangGraph instead of plain Python async?Architecture
LangGraph gives us three things we can't easily replicate in plain Python: (1) Explicit state contracts — every node returns a partial dict that gets merged into ProjectState, so data flow is auditable. (2) Conditional edges — business logic (like "replan if high risks detected") is a named edge, not an if/else buried in a function. (3) Graph visualisation and tracing — the graph is inspectable and the ainvoke call handles the async execution loop for us.
2
How is SharedMemory different from ProjectState?Architecture
They serve different purposes: SharedMemory is a persistent in-process store that lives for the full duration of the server process — agents can read and write it at any point, including across multiple graph runs. It holds the agent summaries, message log, and decision history. ProjectState is the LangGraph state dict that flows through the graph edges — it's the transport layer. Each node reads from it and returns updates. At every node we call _sync_interactions() to copy SharedMemory contents into ProjectState so the final ainvoke() result contains everything.
3
Why does SharedMemory use threading.Lock?Architecture
Python's async model means tasks interleave at every await point. Even though the GIL prevents truly parallel CPU execution, two async tasks can both be "in flight" and both call memory.update_project() nearly simultaneously. Without the lock, one task's update could be overwritten by another. The lock ensures each read-modify-write cycle on the projects dict, messages list, and agent_summaries dict is atomic.
4
Why are routing functions synchronous when everything else is async?Architecture
This is a LangGraph constraint. The graph's internal execution engine calls routing functions synchronously during edge resolution. If they were async def, LangGraph would receive a coroutine object (not a string) and fail. The routing functions are pure state readers — they don't do any I/O, so there's no reason to make them async anyway. All the async work happens inside the node functions.
5
What would you change to scale this to production?Architecture
Three things: (1) Replace in-process SharedMemory with Redis or a proper database — this allows multiple server instances and survives restarts. (2) Add LangGraph's built-in checkpointer (SqliteSaver or PostgresSaver) — this allows workflows to pause, resume, and survive server restarts mid-execution. (3) Remove the MAX_ITERATIONS cap and make it config-driven — for real projects with 50+ tasks, you need dynamic iteration budgets, not hardcoded 3.
Code-Specific Questions
6
Why does AgentRole extend both str and Enum?Code
class AgentRole(str, Enum) makes each enum member a string subclass. This means AgentRole.ENGINEER == "engineer" is True, and it serialises to JSON as "engineer" without calling .value. Without the str mixin, json.dumps({"role": AgentRole.ENGINEER}) would raise TypeError: Object of type AgentRole is not JSON serialisable. This matters because agents store roles in messages and summaries that get JSON-serialised constantly.
7
Why does llm_decide() strip markdown code fences manually?Code
Even when you instruct an LLM to "return only valid JSON, no markdown," GPT models frequently return ```json\n{...}\n```. If we called json.loads() directly on that, it would fail. The stripping logic in llm_decide() handles this in a single place so every agent benefits. The fallback to returning a dict with confidence: 0.65 ensures the workflow degrades gracefully rather than crashing.
8
What is _sync_interactions() and why is it called in every node?Code
_sync_interactions() reads the full current state of SharedMemory (all messages, decisions, feedback loops) and returns it as a dict that nodes include in their state update. It's called in every node because LangGraph's final output is the ProjectState dict returned by ainvoke(). If we only kept data in SharedMemory and never copied it to state, the final result dict would be empty of interaction history. The sync ensures the HTTP response and HTML report contain the complete collaboration log.
9
How does the QA revision loop work at the code level?Code
Four parts work together: (1) _node_qa_critique() calls qa.critique_implementation(), which sets qa_approved and qa_issues in the returned state. (2) _route_after_critique() reads qa_approved and qa_retry_count — if not approved and retries available, returns "revise". (3) _node_engineer_revise() increments qa_retry_count, reads qa_issues from state, and passes them to engineer.revise_implementation(). (4) The linear edge engineer_revise → qa_critique sends execution back to QA. This loop repeats until qa_approved=True or qa_retry_count >= MAX_QA_RETRIES.
10
How does the system handle LLM API failures?Code
Two layers: (1) In compass_integration.py:call_llm(), the entire OpenAI client call is inside a try/except Exception. On any failure (network, auth, timeout), it falls through to _reasoning_fallback(). (2) _reasoning_fallback() uses SHA-256 hash of the input to produce a deterministic, structurally valid JSON response appropriate for each agent type. The workflow never crashes on LLM failure — it continues with fallback data. This means demos work even without live API access.
11
Why does the Engineer use a different file path structure for output?Code
The Engineer writes files to output_examples/{project_id}/spa/. The project_id prefix namespaces outputs per project — multiple projects don't overwrite each other. The spa/ suffix is a convention from the AdminPortalVASMS work this system was designed around (Single Page Application). The engineer creates the full path with mkdir(parents=True, exist_ok=True) so any directory depth returned by the LLM is handled without errors.
12
What is the difference between messages[] and agent_summaries in SharedMemory?Code
messages[] is a chronological event log — point-in-time communications between agents (who said what to whom and when). It's an audit trail. agent_summaries is a key-value store of current state — it represents the latest known output from each agent. When QA reads the Engineer's implementation, it reads agent_summaries["engineer:task_id"] — the most recent Engineer output for that task. If Engineer revises, it overwrites this key. Messages preserve history; summaries reflect current state.
Multi-Agent Design Questions
13
Is this really multi-agent or just one agent broken into functions?Design
Test: remove any one agent and the output changes fundamentally. Remove QAAgent — no revision loop, no quality gate, unvalidated output. Remove RiskAnalyst — no pre-execution replan, critical risks invisible. Remove Stakeholder — no plan approval, no engineer task queue, workflow cannot proceed past planning. Each agent has a unique LLM system prompt, unique SharedMemory key, unique graph nodes, and — critically — reads other agents' stored outputs as input. That cross-agent context dependency is what makes it multi-agent, not multi-function.
14
How do agents "know" what other agents have done?Design
Via SharedMemory.get_summary(role, key). Every agent stores its output after completing a task: memory.store_summary(self.role, project_id, json.dumps(result)). Any other agent can read it with memory.get_summary(target_role, project_id). For example: PM's plan summary is written with key "project_manager:{project_id}". RiskAnalyst reads it with memory.get_summary(AgentRole.PROJECT_MANAGER, project_id) before risk assessment. QA reads Engineer's task summary before critique. This is the primary cross-agent context mechanism.
15
Could agents run in parallel instead of sequentially?Design
Some phases could parallelise, but most can't due to dependencies. RiskAnalyst must wait for PM's plan. Engineer must wait for risk clearance. QA must wait for Engineer. However, mid-execution risk monitoring (after each task) could theoretically run in parallel with the next task's implementation — that's a valid optimisation. LangGraph supports parallel branches with send() — we chose sequential because it's clearer for the hackathon and the dependencies are real, not artificial.
16
Why does the system need a Stakeholder agent? Isn't that just extra validation?Design
The Stakeholder provides something the other agents don't: business perspective independent of technical correctness. QA validates code quality. RiskAnalyst validates delivery risks. Stakeholder validates business value and satisfaction. The approval node in stakeholder_approve is what builds the engineer_task_queue — without it, the execution loop never starts. The satisfaction score also provides a measurable business outcome that Compass evaluates. Removing it collapses the business layer entirely.
17
What happens when an agent produces bad output?Design
Multiple fallback layers: (1) llm_decide() catches JSON parse failures and returns a minimal valid dict with confidence: 0.65. (2) Each agent handles empty/None summaries from SharedMemory with sensible defaults. (3) The graph has defensive code in key nodes — e.g. _node_implement_task() creates a minimal Task if the task_id is missing from the list. (4) If QA max retries are exceeded, it logs a warning and sends a notification to RiskAnalyst, but execution continues. The philosophy: degrade gracefully, never crash.
Collaboration Patterns
18
Describe the QA↔Engineer feedback loop in detail.Collaboration
After Engineer completes a task: (1) Engineer stores implementation summary in agent_summaries["engineer:{task_id}"]. (2) QA reads this + risk context, calls LLM with _CRITIQUE_SYSTEM prompt. (3) LLM returns {"approved": false, "issues": [{"description": "...", "severity": "medium"}]}. (4) QA sends a feedback message to Engineer via provide_feedback(). (5) Graph routing: qa_approved=False + retry_count=0 < 2 → routes to engineer_revise. (6) Engineer reads qa_issues from state, calls LLM with _CODE_REVISE_SYSTEM prompt and the specific issues listed. (7) Engineer rewrites files and stores updated summary. (8) Loops back to QA. (9) QA now uses _RETEST_SYSTEM prompt (different from critique — explicitly checks whether issues were addressed). Up to 2 revision cycles.
19
How does the risk escalation chain work?Collaboration
Two paths: (1) Pre-execution: assess_risks node — RiskAnalyst evaluates PM's plan. If high_severity_count >= 1, sets high_risks_detected=True. Routing sends to replan_project instead of stakeholder_approve. PM reads the risk register summary and asks LLM to adjust tasks accordingly. (2) Mid-execution: After each engineering task, monitor_risks runs. RiskAnalyst calls LLM with current execution context. escalate_risks() checks if any risks are now critical. If yes, sets critical_risks_detected=True, sends an escalation message to PM, routing goes to mid_replan. PM adjusts remaining tasks and the queue continues. Both paths are implemented as conditional edges with named routing functions.
20
What collaboration metrics does the system produce?Collaboration
The _calculate_metrics() method produces: interactions (total messages), feedback_loops (count of unique agent-pair loops), decisions (LLM decision count), by_message_type (dict of type → count, e.g. escalation: 2, feedback: 4), pattern (one of: "collaborative", "iterative_feedback", "delegated_with_escalation"), iterations (engineering loop count). These feed the Compass evaluation API and appear in the HTML report. The pattern classification is automatic: if feedback loops exist → "iterative_feedback"; if escalation messages exist → "delegated_with_escalation".
Hard / Challenging Questions
21
The QA agent critiques a "summary" not actual code. Isn't that just LLM evaluating its own output?Hard
Fair challenge. The system is designed around LLM-to-LLM review — QA uses a different system prompt, different context, and different model invocation than Engineer. The LLM playing QA is told it's a QA engineer reviewing implementation, not writing it. The critic perspective genuinely produces different output from the creator perspective — this is well-established in LLM research. The fallback shows this: ~1/3 of QA calls return rejection with specific issues. In production, QA could also run static analysis tools (eslint, mypy) and feed those results alongside the LLM critique — the architecture supports this via process_task() extension.
22
If the LLM API is unavailable, does the system still demonstrate real collaboration?Hard
Yes — the fallback is input-dependent, not random. The SHA-256 hash of (system_prompt + user_message) determines the output. This means: different agents get different responses (their system prompts differ). The same project input always produces the same trace. The E-Commerce input reliably triggers the high-risk path. QA rejects ~1/3 of calls, Engineer revises, QA re-evaluates. The collaboration pattern is preserved. The fallback is not "return the same thing always" — it's a deterministic simulation of collaboration that exercises all code paths.
23
What prevents an infinite loop in the QA revision cycle?Hard
The routing function: if not approved and retry < MAX_QA_RETRIES: return "revise". MAX_QA_RETRIES = 2. After 2 rejections, the routing function returns "next" regardless of approval — execution continues to monitor_risks. Additionally, _node_qa_critique() detects when max retries are reached and sends a notification to RiskAnalyst: "Task X has unresolved QA issues after 2 retries". This notification becomes part of the final risk report. So unresolved quality issues are not silently ignored — they're escalated.
24
How does the system handle concurrent requests? Can two projects run simultaneously?Hard
Currently, the global memory and orchestrator in run.py are shared across requests. Two simultaneous POST /run calls would share SharedMemory, which could cause project data to intermingle. The workaround: each project is keyed by project_id in memory.projects and all SharedMemory keys include the project ID. In practice, for the hackathon, requests are sequential. For production concurrency, each request should create its own SharedMemory and AgentOrchestrator instance — the CLI path already does this correctly: cli_memory = SharedMemory().
25
Why use gpt-5.1 only for RiskAnalyst and not all agents?Hard
Cost-performance tradeoff. Risk classification is a nuanced reasoning task — "is this risk high or medium for this specific project?" benefits from a more capable model. PM planning, Engineer implementation, and QA critique are structured JSON tasks with clear schemas — gpt-4.1 handles these reliably and faster. Stakeholder approval involves business judgement that a score-based prompt handles well at gpt-4.1 level. The use_reasoning_model=True flag in call_llm() makes this routing explicit and easy to change per agent.
26
What makes this "agentic" rather than just a pipeline of LLM calls?Hard
Three characteristics: (1) Adaptive branching — the workflow path changes based on LLM output (high risks → replan; QA rejection → revision). A pipeline would execute the same steps regardless of what the LLM said. (2) Cross-agent context reading — agents read each other's stored outputs before making decisions. QA reads risk context to know what to scrutinise. PM reads risk register to know what to change when replanning. A pipeline passes data only forward; agents read peer outputs. (3) Escalation — lower-level agents can trigger workflow changes at a higher level. QA at max retries notifies RiskAnalyst. RiskAnalyst mid-execution can trigger PM replanning. These are genuine agentic behaviours, not scripted steps.
27
What's the role of orchestrator.py — why does it exist separately from graph.py?Code
orchestrator.py is a compatibility wrapper written for API stability. run.py calls orchestrator.create_project() and orchestrator.run_workflow(). When we migrated from a simple sequential orchestrator to LangGraph, we kept this API intact by making orchestrator delegate to PMWorkflowGraph. This means run.py didn't need to change. The /status endpoint also uses orchestrator.agents — a dict populated by initialize_agents() that exposes the graph's agent instances for inspection.
28
How does the trace logger work and why JSONL?Code
Every agent action calls trace_logger.log_trace() which appends one JSON object followed by a newline to logs/agent_trace.jsonl. JSONL (JSON Lines) is chosen because: (1) It's append-only — no file locking complexity, no reading the whole file to add an event. (2) Each line is independently parseable — if the process crashes mid-run, all prior events are intact. (3) Tools like jq, pandas, or any log aggregator (Splunk, ELK) can stream-process it without loading the whole file. Each event includes: timestamp, trace_id (shared by all events in a run), agent_name, action, input/output summaries, confidence, retry_count, and target_agent.
29
Why is temperature set to 0.3 for all LLM calls?Design
These agents produce structured JSON, not creative text. Higher temperature increases JSON structure variability — missing fields, extra keys, different naming — which causes parse failures. Lower temperature (0.0-0.3) gives more deterministic, schema-adherent output. 0.3 rather than 0.0 preserves enough variability that different project descriptions get qualitatively different plans (rather than always the same plan). It's the minimum needed for useful domain variation without sacrificing structural reliability.
30
How does the system know which agent is calling the fallback?Code
The fallback detects the calling agent by reading specific phrases in the system prompt: "qa engineer ai agent" → QA agent, "risk analyst ai agent" → RiskAnalyst, "project manager ai agent" → PM, etc. Each agent's system prompt contains its role description as the opening phrase. This is intentional — it makes fallback detection reliable without requiring agents to pass their identity separately. The alternative (passing agent_name to call_llm()) was considered but rejected: the system prompt is always available, so no extra parameter needed.

What real public data does the system fetch, and why?Architecture

How does the Engineer generate the MVP, and why two-stage extraction?Code
The Engineer calls the LLM directly with compass.call_llm(_MVP_SYSTEM, context, max_tokens=4000) — bypassing llm_decide() which would truncate at 1,200 tokens and break large HTML files. The two-stage extraction handles LLM response format variance: Stage 1 — attempt json.loads() on the response, looking for a dict with key "html_content" (the structured case). Stage 2 — if Stage 1 fails or html_content doesn't start with <!, run a re.search(r"(<!DOCTYPE\s+html.*?</html>)", raw, re.DOTALL) regex to extract a raw HTML block. Without Stage 2, raw HTML responses produce an empty MVP and the frontend shows no MVP link. The max_tokens=4000 budget is required because a self-contained HTML dashboard with CSS and JS needs 2,000–4,000 tokens.

How is report generation made reliable even when the workflow throws an exception?Code
Two exception layers in app/graph.py: (1) The _app.ainvoke() call is wrapped in a broad except Exception as exc block — if any graph node raises (not just asyncio.TimeoutError as before), final_state = state is set so the partial result dict is preserved. (2) The generate_report(final_state, project_name) call is wrapped in its own try/except Exception — if HTML rendering fails (e.g. Mermaid template error), the exception is logged and result["report_path"] = "" is set, so the FastAPI handler still receives a valid response dict and can return HTTP 200 with a JSON body.

How are interaction_counts and collaboration_patterns computed?Code
interaction_counts is built in the POST /run handler by counting trace event actions that match specific keywords: _count("plan") counts actions containing "plan", _count("implement") counts "implement", etc. This produces a dict like {"plan": 3, "implement": 4, "qa": 6, "risk": 5} that gives judges a breakdown of where agent effort went. collaboration_patterns is detected by collecting the action set from all trace events and checking for keyword presence: if any action contains "escalate" or "replan" → "risk_escalation_to_pm"; if any action contains "needs_revision" or "critique" → "qa_critique_and_revision_loop"; if any action contains "replan" → "replan_based_on_feedback". The patterns list is dynamic — it reflects what actually happened in that run, not a static label.

Why does the frontend need cache busting with ?v=N on CSS and JS URLs?Code
Browsers aggressively cache static files. When frontend/dist/app.js or styles.css changes, browsers continue serving the old cached version until either the cache expires or the URL changes. Adding ?v=6 (or any query parameter) to the script/link tag URL makes the browser treat it as a different resource — the old cached version is ignored and the new file is fetched. This is a standard cache-busting technique. In this project, the version number is incremented manually in frontend/dist/index.html whenever JS or CSS changes. Without this, judges would see "Project: undefined, Iterations: undefined" even after deploying updated code, because their browser loads the old app.js that reads from the old response format.

What are the Compass model identifiers and which agent uses which?Architecture
Two model tiers: gpt-5.1 (reasoning model, use_reasoning_model=True) and gpt-4.1 (standard model). Only RiskAnalyst uses gpt-5.1 — risk classification is a nuanced judgement task that benefits from deeper reasoning. All other agents (PM, Engineer, QA, Stakeholder) use gpt-4.1 — they produce structured JSON against a defined schema, which gpt-4.1 handles reliably at lower latency and cost. The model selection is exposed in the RunResponse under metadata.compass_models: {"reasoning": "gpt-5.1", "standard": "gpt-4.1"} so judges can see that the system uses differentiated models, not one model for everything.

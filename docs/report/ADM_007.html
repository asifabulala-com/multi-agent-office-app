Project Plan

Design and implement a digital Visitor Management System for a telecom company to replace paper sign-in books, enabling pre-registration, kiosk self-check-in, ID scanning, badge printing, NDA e-signature, host notifications, and evacuation list management. The system will integrate with existing access control systems and Active Directory for host lookup, and provide visit logs, contractor tracking, and compliance reporting for physical security audits. Solution must run on Raspberry Pi-based kiosks with a secure, responsive React frontend and a Python backend.

Total effort: 218hRecommended team: 5 peoplePhases: 3

FrontendReact 18 + TypeScript
DatabasePostgreSQL 15 + Redis 7 for caching
InfrastructureDocker + Kubernetes (on AWS EKS) + NGINX Ingress

task_001 Detailed requirements, data model, and system architecture design MEDIUM 20h
Work with stakeholders (physical security, IT, facilities) to capture detailed functional and non-functional requirements, define user roles (receptionist, host, security admin, contractor manager), and design the overall system architecture. Produce an ERD for visitors, hosts, visits, NDAs, badges, access control tokens, and audit logs. Define integration patterns with Active Directory, access control systems (via REST or message queue), and notification providers (SMS/email). Identify privacy, data retention, and security constraints relevant to telecom environments.
Acceptance Criteria
  • Approved requirements specification document covering all listed features (pre-registration, kiosk check-in, ID scanning, badge printing, NDA e-signature, host notifications, evacuation lists, contractor tracking, compliance reporting).
  • ERD and API high-level design reviewed and signed off by lead engineer and security representative.
  • Architecture diagram (including Raspberry Pi kiosks, backend services, database, and external integrations) stored in the project repository (e.g., as a draw.io or PlantUML file).
  • Documented non-functional requirements including performance (e.g., <2s check-in response), availability targets, and data retention policies for visit logs.
Deliverables
  • Requirements specification document (PDF/Confluence page).
  • Entity-Relationship Diagram (ERD) for core data model.
  • System architecture diagram with integration points.
  • Risk and compliance considerations summary for physical security audits.

task_002 Implement core backend services and database schema MEDIUM 28h
Set up FastAPI project structure with modular routers for visitors, hosts, visits, NDAs, and authentication. Configure PostgreSQL with SQLAlchemy ORM and Alembic migrations. Implement core CRUD APIs for visitors, hosts, and visits, including contractor flagging and visit types (guest, contractor, vendor). Implement JWT-based authentication and role-based access control for admin and receptionist portals. Ensure all endpoints are documented with OpenAPI and protected appropriately.
Acceptance Criteria
  • FastAPI application runs locally via Docker Compose with PostgreSQL and Redis containers.
  • Alembic migrations create tables for visitors, hosts, visits, NDAs, badges, and audit logs as per ERD.
  • REST endpoints exist for creating and querying visitors, hosts, and visits, with input validation using Pydantic models.
  • OpenAPI docs available at /docs and show all core endpoints with correct request/response schemas.
  • Unit tests cover at least 70% of core service logic (visitor/visit creation, role checks) and pass in CI.
Deliverables
  • FastAPI backend repository with configured project structure.
  • Alembic migration scripts for initial schema.
  • API documentation via OpenAPI/Swagger UI.
  • Unit test suite and CI pipeline configuration (e.g., GitHub Actions).

task_003 Integrate with Active Directory and notification providers HIGH 18h
Implement integration with the company's Active Directory for host lookup and authentication using python-ldap or an appropriate AD/LDAP client. Create an API endpoint to search hosts by name or email, restricted to authenticated users. Configure Twilio for SMS and SendGrid (or company SMTP) for email notifications. Implement a notification service that sends host notifications on visitor check-in and supports templated messages.
Acceptance Criteria
  • Backend can connect to a test AD/LDAP server and successfully query user objects by name and email.
  • Host search API returns AD user attributes (name, email, department) in under 1 second for typical queries.
  • Twilio and email provider credentials are read from environment variables and not hard-coded.
  • When a visit is created with a host, a test endpoint can trigger SMS and email notifications to a configurable phone number and email address.
  • Integration tests simulate host lookup and notification sending and pass in CI.
Deliverables
  • AD integration module with configuration documentation.
  • Notification service module supporting SMS and email templates.
  • Host lookup API endpoint with authentication.
  • Integration test suite for AD and notification flows.

task_004 Develop React web portal for reception and security staff MEDIUM 30h
Build a React 18 + TypeScript web application for receptionists and security admins. Implement login, dashboard showing current on-site visitors, manual visitor registration, pre-registration management, contractor tracking, and visit search. Use Material UI for layout and React Query for API data fetching. Integrate host lookup via AD-backed API and allow receptionists to create visits linked to hosts. Include role-based UI elements for security admins to view audit logs and configure NDA templates.
Acceptance Criteria
  • Receptionist can log in and see a dashboard listing all visitors currently checked in with visit details.
  • Receptionist can pre-register a visitor by entering name, email, company, visit date/time, and selecting a host via AD lookup.
  • Contractor visits can be flagged and filtered in the visit search view.
  • Security admin role can access a configuration section to manage NDA templates (create/update text) and view audit logs.
  • All API calls use React Query with loading/error states and are protected by JWT tokens stored in memory (not localStorage).
Deliverables
  • React web portal codebase integrated with backend APIs.
  • Role-based navigation and views for receptionist and security admin.
  • Pre-registration and visit management UI flows.
  • Basic styling and responsive layout suitable for desktop browsers.

task_005 Implement Raspberry Pi kiosk app for visitor self-check-in HIGH 34h
Create a kiosk-oriented React frontend optimized for full-screen Chromium on Raspberry Pi, providing visitor self-check-in and check-out. Implement flows for scanning a pre-registration QR code, manual entry of visitor details, capturing NDA e-signature using react-signature-canvas, and printing visitor badges via a connected thermal printer using a lightweight Node.js or Python service on the Pi (e.g., node-thermal-printer). Integrate optional ID scanning via USB scanner as keyboard wedge input. Ensure the kiosk app auto-starts on boot and runs in kiosk mode.
Acceptance Criteria
  • Kiosk app runs in full-screen mode on Raspberry Pi OS with Chromium and auto-launches on boot.
  • Visitor can scan a pre-registration QR code and complete check-in in under 60 seconds, including NDA signature and badge printing.
  • Manual check-in flow allows entry of name, company, host selection, and visit reason, and records NDA acceptance with stored signature image.
  • Badge printing produces a legible badge with visitor name, host, company, visit type, and expiry time using the configured thermal printer.
  • Check-out flow allows visitors to scan their badge QR code or search by name to mark departure, updating the backend in real time.
Deliverables
  • Kiosk React app bundle configured for Raspberry Pi deployment.
  • Raspberry Pi setup scripts (systemd service or autostart configuration) for kiosk mode.
  • Local print service (Node.js or Python) to interface with the thermal printer.
  • Documentation for kiosk installation and configuration (including supported printer models).

task_006 Access control system integration and time-limited access tokens HIGH 26h
Integrate the visitor management backend with the building's access control system (e.g., via REST API or message queue) to issue time-limited access credentials when a visitor checks in. Implement a configurable mapping between visit types and access levels. Generate access tokens or barcodes encoded on visitor badges and ensure automatic revocation or expiry at the end of the visit. Log all access control operations for audit purposes.
Acceptance Criteria
  • Configuration exists to map visit types (guest, contractor, vendor) to access control roles or zones.
  • On successful check-in, the system calls the access control API (or publishes a message) to create a temporary credential with an expiry matching the scheduled visit end time.
  • Badge QR or barcode encodes a token or ID that the access control system can validate at entry points.
  • On check-out or visit expiry, the system revokes or deactivates the associated access credential and logs the event.
  • Integration tests using a mocked access control API verify credential creation, update, and revocation flows.
Deliverables
  • Access control integration module with configuration for different environments.
  • Extended visit and badge models to store access control identifiers and expiry.
  • Audit log entries for all access control operations.
  • Integration test suite and configuration documentation for connecting to the real access control system.

Deliverables

  • Compliance reporting UI and backend endpoints.
